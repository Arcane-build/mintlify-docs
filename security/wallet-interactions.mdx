## Permissions requested

| Interaction | Requested? | Notes |
| --- | --- | --- |
| Connect wallet (read public key) | ✅ Yes | Via Privy, Solana-only; external wallets |
| Sign-in message (`signMessage`) | ✅ Yes | Human-readable, prefixed `Microsets:wallet-auth:`; off-chain; authorizes a backend JWT only |
| Sign transaction (`signTransaction` / `sendTransaction`) | ✅ Yes | Only on explicit user action (trade, add/withdraw liquidity, claim) |
| Embedded wallet creation | ❌ No | Disabled (`createOnLogin: 'off'`) |
| Auto-approve / blind signing | ❌ No | [TODO: confirm no auto-approve is enabled in production] |

## Signature types

<AccordionGroup>
  <Accordion title="Message signing (off-chain)">
    Used once per connected wallet for authentication. The signed payload begins
    with `Microsets:wallet-auth:` and is verified server-side to issue a JWT. It
    grants no on-chain authority and moves no funds.
  </Accordion>
  <Accordion title="Transaction signing (on-chain)">
    Used for `swap`, `add_liquidity`, `withdraw_liquidity`, and `claim_winnings`.
    Each is a standard Solana transaction the user reviews in their wallet.
  </Accordion>
  <Accordion title="Session wallet delegate">
    Optionally, a short-lived session keypair is derived from a user-signed
    message so repeated bets don't re-prompt. [TODO: document the session key's

  </Accordion>
</AccordionGroup>

## Summary

Microset requests a wallet connection, one human-readable sign-in message, and
explicit transaction signatures for each trade or claim. Outcome-token supply is
fixed (mint authority revoked), and funds move only through documented
instructions. Complete the `[TODO]` rows — especially the session-wallet scope and
