Every integration in Microset authenticates with the external service using one of three methods: API keys, OAuth 2.0, or service accounts. The connector catalog specifies which method each connector supports. Understanding how each method works helps you choose the right setup and keep your credentials secure.

## API keys

An API key is a long-lived token issued by the external service that identifies your account for every request Microset makes on your behalf. API keys are the simplest authentication method and work well for server-to-server connections.

**When to use API keys:** Choose this method for services that issue personal access tokens or API keys — such as developer tools, data APIs, and analytics platforms — where you don't need delegated user authorization.

### Rotate an API key

<Info>
  Rotating credentials regularly limits the blast radius if a key is ever exposed. Most security best practices recommend rotating API keys at least once per year, or immediately after any suspected leak.
</Info>

<Steps>

<Step title="Generate a new key in the external service">
Sign in to the external service and navigate to its developer or API settings. Generate a new API key or personal access token. Copy the key — most services only show it once.
</Step>

<Step title="Open the connector credentials in Microset">
In Microset, go to **Integrations** → select the integration by name → **Credentials**.
</Step>

<Step title="Paste the new key and save">
Clear the existing key field, paste your new key, and click **Save**. Microset immediately begins using the new key for all outbound requests.
</Step>

<Step title="Verify, then revoke the old key">
Trigger a manual sync or click **Test connection** to confirm the integration is working correctly with the new key. Once you've verified the connection, return to the external service and revoke or delete the old key.
</Step>

</Steps>

<Warning>
  Do not revoke your old API key before saving the new one in Microset. If you revoke first, all syncs will fail until the new key is in place. Save the new key in Microset first, verify the connection works, and only then revoke the old key in the external service.
</Warning>

## OAuth 2.0

OAuth 2.0 lets you authorize Microset to act on your behalf within an external service without sharing your password. You grant access through the service's own authorization screen, and Microset stores the resulting tokens securely. Microset handles token refresh automatically, so you don't need to re-authorize on a schedule.

**When to use OAuth 2.0:** Choose this method for SaaS tools that support OAuth — including CRMs, productivity suites, communication platforms, and calendar or file-storage services.

### How the OAuth flow works

<Steps>

<Step title="Initiate authorization">
On the connector setup page in Microset, click **Authorize**. Microset constructs the authorization request and redirects your browser to the external service's authorization screen.
</Step>

<Step title="Sign in to the external service">
Sign in to the external service using your account credentials if you aren't already signed in.
</Step>

<Step title="Approve the requested permissions">
Review the list of permissions Microset is requesting — such as reading contacts or writing calendar events — and click **Allow** or **Approve** to grant access.
</Step>

<Step title="Return to Microset">
The service redirects your browser back to Microset automatically. Microset stores the access token and refresh token securely in encrypted storage. The connector is now active.
</Step>

</Steps>

<Note>
  OAuth access tokens expire regularly (typically every hour). Microset refreshes them automatically in the background using the stored refresh token, so syncs continue without interruption. If the refresh fails — for example, because a workspace admin revoked Microset's access in the external service — the integration status changes to **Reconnect required**. Open the connector in Microset and click **Reconnect** to re-authorize.
</Note>

## Service accounts

A service account is a non-personal identity created specifically for machine-to-machine access. Unlike API keys tied to an individual user account or OAuth tokens tied to a person's login, service accounts exist independently of any employee — so access doesn't break when someone leaves your organization.

**When to use service accounts:** Use a service account when:

- Access must not be tied to an individual user (for compliance, continuity, or security reasons)
- The external service doesn't support OAuth 2.0
- You're connecting to infrastructure services that expect machine credentials

Service accounts are most common with **Google Cloud** (JSON key files), **AWS** (IAM roles and access keys), and **enterprise SaaS platforms** with admin-level API access.

### Set up a service account

Upload your credentials file in the connector setup dialog. The file format depends on the service:

| Service type | Typical format |
|---|---|
| Google Cloud / Workspace | JSON key file exported from the GCP console |
| AWS | Access key ID and secret access key (entered as fields, not a file) |
| Enterprise SaaS | JSON key or PEM certificate, depending on the provider |

Microset stores the uploaded credentials file using encryption at rest. Fill in any additional required fields shown in the dialog (such as a project ID or domain), then click **Save**.

### Rotate service account credentials

<Tip>
  Schedule credential rotations in your team's calendar so they never slip — many compliance frameworks require rotation every 90 days.
</Tip>

1. Generate a new key or certificate in the external service's admin console. Keep the old key active until the rotation is complete.
2. In Microset, go to **Integrations** → select the integration → **Credentials**, and upload the new credentials file.
3. Click **Test connection** to verify Microset can authenticate with the new credentials.
4. Once the test passes, deactivate or delete the old key in the external service.
